# PurpleTrace Investigation Workflow

PurpleTrace Investigation Workflow explains how to review alerts generated by PurpleTrace Agent.

The goal is to move from a raw detection alert to a structured analyst review.

---

## Workflow Overview

A PurpleTrace investigation should answer these questions:

1. What rule generated the alert?
2. What process or command triggered the detection?
3. Which MITRE ATT&CK tactic and technique are mapped?
4. What evidence matched the rule?
5. What source event produced the alert?
6. What should an analyst review next?
7. Is this expected activity, suspicious activity, or a false positive?

---

## Investigation Steps

### 1. Identify the Alert

Start with the basic alert fields:

* Alert ID
* Timestamp
* Rule ID
* Rule name
* Severity
* Hostname
* Username

These fields explain when the alert happened and which rule triggered it.

---

### 2. Review Rule Metadata

Review the detection rule metadata:

* Rule ID
* Rule name
* Rule severity
* Rule author
* Rule creation date
* Rule tags
* Rule references

This helps understand the purpose and scope of the detection.

---

### 3. Map to MITRE ATT&CK

Review the MITRE mapping:

* Tactic
* Technique ID
* Technique name

This explains the behavior category.

Example:

```text
Tactic: Execution
Technique: T1059.001 - PowerShell
```

---

### 4. Review Process Evidence

Check process-related fields:

* Process name
* Process path
* Command line
* Parent process name
* Parent command line

This helps understand what ran and how it was started.

---

### 5. Review Matched Evidence

PurpleTrace alerts include matched evidence fields:

* Evidence summary
* Matched fields
* Matched values

Example:

```text
EvidenceSummary:
ProcessName contains powershell.exe; CommandLine contains -NoProfile
```

This shows exactly why the rule matched.

---

### 6. Review the Source Event

The source event is the original endpoint telemetry object that caused the alert.

Important fields:

* Event source
* Event ID
* Event type
* Hostname
* Username
* Process ID
* Process name
* Command line
* Parent process
* Additional data

The source event should be reviewed to confirm the alert context.

---

### 7. Decide Investigation Outcome

The analyst should classify the alert as one of the following:

* Expected activity
* Suspicious activity
* Needs more context
* False positive

---

## Recommended Analyst Questions

Use these questions during review:

* Was the command expected for this user?
* Was the parent process normal?
* Is the command line suspicious?
* Does the activity match normal admin behavior?
* Is the MITRE mapping appropriate?
* Are there repeated alerts from the same host?
* Are multiple rules firing on the same event?
* Does the evidence summary explain the detection clearly?

---

## Example Investigation Flow

```text
Alert:
PT-RULE-001 Suspicious PowerShell Execution

MITRE:
T1059.001 - PowerShell

Evidence:
ProcessName contains powershell.exe
CommandLine contains -NoProfile
CommandLine contains ExecutionPolicy Bypass

Analyst Review:
PowerShell was launched from cmd.exe with suspicious command-line flags.
The activity should be reviewed to determine whether it was expected admin behavior or suspicious execution.
```

---

## Safety Scope

This workflow is for defensive alert review.

It does not include:

* Exploit development
* Malware execution
* Credential theft
* Persistence creation
* Evasion implementation
* Offensive payloads
